| Estado        | ✅ Completo                                                                           |

---

## Agente

- **role**: Ingeniero de QA / validación de módulos.
- **goal**: Verificar que `build_reporte_umata_llm()` y `render_pdf()` mejorado funcionan correctamente, que el fallback opera cuando el LLM no está disponible, y que la retrocompatibilidad con `build_reporte()` se mantiene.
- **backstory**: Cada pipeline del proyecto tiene su script de validación. C3.3 sigue el mismo patrón que `validate_a3.py`, `validate_m4.py`, etc.

---

## Objetivo

Crear `specs/C3_reportes/validate_c3.py` con una función `run_validations()` que ejecute
todas las validaciones de C3 y reporte resultados por consola. El script debe poder ejecutarse
directamente con `python specs/C3_reportes/validate_c3.py` desde la raíz del proyecto.

---

## Alcance

### Incluye

- Validación de `build_reporte()` (retrocompatibilidad — sin cambios de firma ni esquema).
- Validación de `build_reporte_umata_llm()` con LLM disponible (si `LLM_API_KEY` está configurada).
- Validación de `build_reporte_umata_llm()` con LLM simulado como fallback (mock de excepción).
- Validación de `render_pdf()` con dict de `build_reporte()`.
- Validación de `render_pdf()` con dict de `build_reporte_umata_llm()`.
- Verificación del semáforo: el PDF contiene el color correcto según el nivel de riesgo.
- Verificación de la tabla de indicadores: el PDF tiene tamaño > 1 KB.
- Verificación de que `render_pdf()` no lanza excepción con dict sin `prediccion_riesgo`.
- Reporte final: número de validaciones pasadas / total.

### No incluye

- Tests de carga o rendimiento.
- Validación del endpoint HTTP (eso es responsabilidad de `validate_a3.py`).
- Comparación visual del PDF (solo verificación de bytes y tamaño).

## Lógica / Proceso sugerido

```
run_validations()
├── V1: Retrocompatibilidad build_reporte()
│   ├── Llamar build_reporte("73168", "Chaparral", "Tolima", "Café")
│   ├── Verificar que retorna dict con claves: codigo_dane, municipio, departamento,
│   │   cultivo, año_referencia, titulo, contenido_texto, secciones, fuentes
│   └── PASS / FAIL con mensaje
│
├── V2: Esquema de build_reporte_umata_llm() — sin LLM (mock fallback)
│   ├── Parchear get_chat_engine() para que lance Exception
│   ├── Llamar build_reporte_umata_llm("73168", "Chaparral", "Tolima", "Café")
│   ├── Verificar que retorna dict válido con generado_por_llm=False
│   └── PASS / FAIL
│
├── V3: build_reporte_umata_llm() con LLM real (solo si LLM_API_KEY disponible)
│   ├── Si no hay LLM_API_KEY → SKIP con mensaje
│   ├── Llamar build_reporte_umata_llm("73168", "Chaparral", "Tolima", "Café")
│   ├── Verificar generado_por_llm=True y contenido_texto no vacío
│   └── PASS / FAIL / SKIP
│
├── V4: render_pdf() con dict de build_reporte()
│   ├── Llamar render_pdf(build_reporte("73168", "Chaparral", "Tolima", "Café"))
│   ├── Verificar que retorna bytes
│   ├── Verificar que len(bytes) > 1024 (> 1 KB)
│   ├── Verificar que los primeros 4 bytes son b'%PDF'
│   └── PASS / FAIL
│
├── V5: render_pdf() con dict de build_reporte_umata_llm() (fallback)
│   ├── Usar el dict del V2 (generado_por_llm=False)
│   ├── Llamar render_pdf(dict_v2)
│   ├── Verificar bytes válidos > 1 KB
│   └── PASS / FAIL
│
├── V6: render_pdf() sin clave prediccion_riesgo (retrocompatibilidad)
│   ├── Construir dict mínimo sin prediccion_riesgo
│   ├── Llamar render_pdf(dict_minimo)
│   ├── Verificar que no lanza excepción y retorna bytes válidos
│   └── PASS / FAIL
│
├── V7: ValueError para municipio sin predicción
│   ├── Llamar build_reporte_umata_llm con municipio válido pero cultivo sin datos
│   │   (usar un cultivo que no tenga registros para ese municipio)
│   ├── Verificar que lanza ValueError
│   └── PASS / FAIL
│
└── Resumen: "C3 Validaciones: X/7 pasadas"
```

## Manejo de errores

| Situación                                          | Comportamiento en el script                                                    |
| -------------------------------------------------- | ------------------------------------------------------------------------------ |
| Parquet no encontrado                              | Marcar V1–V7 como FAIL con mensaje `"Parquet no encontrado: {path}"`           |
| `LLM_API_KEY` no configurada                       | V3 → SKIP con mensaje `"[SKIP] LLM_API_KEY no configurada"`                   |
| Excepción inesperada en una validación             | Capturar, loggear, marcar como FAIL y continuar con la siguiente               |
| `build_reporte_umata_llm` no existe en reports.py  | Marcar V2, V3, V5 como FAIL con `"ImportError: función no implementada"`       |

## Caso feliz

- **Dado** que los parquets existen y `LLM_API_KEY` está configurada.
- **Cuando** se ejecuta `python specs/C3_reportes/validate_c3.py`.
- **Entonces** imprime `C3 Validaciones: 7/7 pasadas` y termina con código de salida 0.

## Validación

- **Dado** que `LLM_API_KEY` no está configurada.
- **Cuando** se ejecuta el script.
- **Entonces** V3 aparece como SKIP, el resto pasa, y el script termina con código 0.

- **Dado** que `build_reporte_umata_llm` no está implementada.
- **Cuando** se ejecuta el script.
- **Entonces** V2, V3 y V5 aparecen como FAIL y el script termina con código 1.

---

## Criterios de aceptación

1. El script existe en `specs/C3_reportes/validate_c3.py` y es ejecutable desde la raíz del proyecto.
2. `run_validations()` retorna `True` cuando todas las validaciones pasan (o son SKIP).
3. Cada validación imprime `[Vn] Descripción    PASS | FAIL | SKIP` con alineación legible.
4. El script termina con código de salida 0 si todas pasan/skip, 1 si alguna falla.
5. No modifica ningún archivo de datos ni de código del proyecto.
6. Funciona sin conexión a internet si `LLM_API_KEY` no está configurada (V3 → SKIP).
