# Spec: F3.1 Pantalla de Comparador de Cultivos

## Metadatos

- ID: F3.1
- Módulo o área: Frontend (Comparador)
- Depende de: Entregables de F2 (Ficha municipal base), endpoints del A1/A3
- Produce para: Usuario final
- Estado: ✅ Completo

## Agente

- role: Frontend Developer experto en React, Next.js y Tailwind CSS.
- goal: Crear una vista interactiva que permita comparar la viabilidad de sembrar Café, Cacao o Maíz en un municipio específico.
- backstory: Tu solución es la clave para la "traducción" de modelos de IA en consejos accionables. Diseñarás un comparador que un evaluador de UMATAs usará para aconsejar a los campesinos qué cultivo sembrar.

## Objetivo

- Renderizar una pantalla interactiva con un ranking visual de recomendación de los 3 cultivos MVP y una tabla cruzando métricas predictivas y variables de riesgo, ofreciendo un soporte analítico claro.

## Alcance

### Incluye

- Creación de una ruta o componente Next.js (`ComparadorCultivos`).
- Tarjetas de "Ranking" estilizadas (1er, 2do y 3er puesto).
- Tabla comparativa detallada mostrando: Rendimiento esperado, Riesgo climático, Riesgo económico y Nivel de aptitud (UPRA).
- Integración de badges tipo semáforo (`shadcn/ui`) para visualizar el estado de cada métrica.
- Componente de estado de "Cargando" (skeleton loaders) mientras se obtienen datos.

### No incluye

- Implementación de la lógica del backend para calcular el _score_ final (eso se entrega listo por la API A3/A1).
- Mapas geográficos (Manejado en F1).

## Entradas

- El ID del municipio (código DANE) seleccionado por el usuario en la interfaz global (puede venir vía URL o Context).

## Inputs esperados

- Datos provenientes de la API backend (JSON de respuesta a `POST /predecir` iterado o a un nuevo `GET /recomendar/{municipio}`).
- Estructura base inferida del modelo:
  ```json
  [
    {
      "cultivo": "Café",
      "rendimiento_esperado_t_ha": 1.2,
      "riesgo_climatico": "Bajo",
      "riesgo_economico": "Medio",
      "aptitud_upra": "Alta",
      "score_recomendacion": 85
    },
    ...
  ]
  ```

## Lógica / Proceso sugerido

1. El usuario selecciona el municipio objetivo.
2. El componente activa un fetch hacia la API para las proyecciones de los cultivos MVP (café, cacao, maíz).
3. Una vez retornada la información, se ordena del `score_recomendacion` más alto al más bajo.
4. **Ranking Visual**: renderizar un podio o sección de tarjetas ordenadas. Ejemplo: "🏆 #1 Café", "🥈 #2 Cacao", etc.
5. **Tabla de Resultados**: renderizar una fila por cultivo desglosando los valores específicos, donde las columnas de "riesgo" aplican badges semaforizados (Rojo=Alto_Riesgo, Amarillo=Medio_Riesgo, Verde=Bajo_Riesgo_o_Aptitud_Alta).
6. Si ocurre un fallo en la solicitud, presentar un `Alert` de error limpio usando la misma UI.

## Interfaz o contrato

- Frontend Route: `frontend/app/comparador/page.tsx`
- Componentes React modulares (ej. `RankingCard`, `ComparisonTable`).

## Manejo de errores

- **Fetch fallido**: Mostrar texto "Error conectando con el orquestador. Intenta más tarde."
- **Datos insuficientes**: Si falta el histórico de un cultivo en el municipio provisto, en su score mostrar "N/A" y en la fila advertir "Sin suficiente histórico para proyectar".

## Salida esperada

- Carpetas de código cliente para el frontend que cumplan el contrato visual.
- Testing rápido visual: los semáforos funcionan dependiendo del nivel y las cards se organizan solos.

## Caso feliz

- Dado un municipio como Ibagué (73001)
- Cuando el usuario accede al componente de Comparador
- Entonces ve las proyecciones en 3 tarjetas que dan la recomendación prioritaria a Cacao, seguido de Maíz y por último a Café, explicados en una tabla semaforizada en tonos verdes, limpios y accesibles.

## Validación

- Dado un municipio en donde no es apto (UPRA Baja o Restricción) sembrar uno de los cultivos.
- Cuando el usuario pide la comparación.
- Entonces el cultivo descartado cae al tercer lugar y aparece el riesgo indicado con un semáforo rojo.

## Criterios de aceptación

- Claridad técnica y visual usando directrices `shadcn/ui`.
- Atomicidad del componente del comparador.
- Trazabilidad y correcta gestión de estados vacíos e invalidaciones.

## Decisiones

| Fecha | Decisión | Motivo | Impacto |
|-------|----------|--------|---------|
| 2026-05-28 | Score calculado en frontend como `(1 - prob_riesgo_alto) * 60 + (rend/maxRend) * 40 - 40` | No existe endpoint `/recomendar` en el backend; se usa `POST /predecir` para los 3 cultivos | Score relativo al municipio, no absoluto |
| 2026-05-28 | Municipios cargados desde `GET /municipios` sin hardcode | Criterio de no hardcodear datos en frontend | `MunicipioSelector` recibe lista dinámica |
| 2026-05-28 | Ruta `/comparador?municipio=&anio=` con query params | Permite deep-linking y navegación desde el mapa | `page.tsx` lee `searchParams` |
