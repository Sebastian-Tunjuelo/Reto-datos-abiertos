# Spec: M5.1 - Función de escenarios

## Metadatos

- ID: M5.1
- Módulo o área: Predictivo / simulación de escenarios
- Depende de: M1, M2, M3
- Produce para: A3 (`POST /escenario`) y la interfaz de comparación de escenarios
- Estado: Completado

## Agente

- role: Python backend engineer
- goal: Implementar un motor determinista de escenarios contrafactuales sobre la fila base de inferencia
- backstory: Debe convertir shocks simples en una comparación reproducible entre línea base y escenarios sin tocar el entrenamiento de los modelos

## Objetivo

- Generar escenarios contrafactuales de rendimiento y riesgo a partir de una fila base de features, aplicando el catálogo canónico `base`, `seco`, `lluvioso` y `fertilizantes`, y devolver una comparación clara entre la línea base y cada shock.

## Alcance

### Incluye

- Validar una fila base de inferencia con el esquema esperado por los modelos M2 y M3.
- Construir variantes de esa fila para los escenarios canónicos del MVP.
- Aplicar shocks deterministas y limitados a un allowlist de features.
- Ejecutar los modelos de regresión y clasificación sobre cada variante.
- Comparar cada escenario contra la línea base y calcular deltas.
- Exponer una salida estructurada lista para el endpoint `POST /escenario`.

### No incluye

- Reentrenar modelos o recalibrar umbrales.
- Generar pronóstico meteorológico real.
- Explorar escenarios fuera del catálogo canónico.
- Generar narrativas LLM o explicabilidad SHAP en este módulo.
- Hacer búsqueda automática del mejor escenario.

## Entradas

- Una fila base de features alineada con el esquema de inferencia de M1.
- Artefactos de M2 y M3 para el cultivo solicitado.
- Metadatos de features para restaurar el orden exacto esperado por cada modelo.
- Un catálogo de escenarios solicitado por el caller, limitado a `base`, `seco`, `lluvioso` y `fertilizantes`.

## Inputs esperados

| Input                  | Reglas mínimas                                                                                                                                                                                      |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Fila base              | Debe contener `codigo_dane`, `municipio`, `departamento`, `cultivo`, `año` y todas las features numéricas usadas por M2/M3. Puede venir de `data/feature_matrix.parquet` o de la capa orquestadora. |
| Modelo regresor        | Un `.pkl` por cultivo con su `*_meta.json` asociado.                                                                                                                                                |
| Clasificador de riesgo | Un `.pkl` por cultivo con su `*_meta.json` asociado.                                                                                                                                                |
| Catálogo de escenarios | Solo valores reconocidos por el contrato global. `base` debe estar disponible como referencia.                                                                                                      |

### Variables que sí puede perturbar

| Escenario       | Variables permitidas                                                      | Observación                                                                      |
| --------------- | ------------------------------------------------------------------------- | -------------------------------------------------------------------------------- |
| `seco`          | `prec_acum_mm`, `anomalia_prec`, `dias_secos`                             | La transformación debe ser monotónica y determinista.                            |
| `lluvioso`      | `prec_acum_mm`, `anomalia_prec`, `dias_secos`                             | La transformación debe ser monotónica y determinista.                            |
| `fertilizantes` | `indice_agroinsumos`, `percentil_fertilizantes`, `señal_riesgo_economico` | Las columnas derivadas deben seguir una regla local, determinista y documentada. |

Los escenarios de clima y economía deben operar sobre las columnas que realmente existen en la fila base: `prec_acum_mm`, `anomalia_prec`, `dias_secos`, `indice_agroinsumos`, `percentil_fertilizantes` y `señal_riesgo_economico`.

### Variables que no debe usar como entrada

- `target_rendimiento`
- `produccion`
- `area_cosechada`
- Cualquier identificador que no forme parte del esquema de inferencia del modelo

## Lógica / Proceso sugerido

1. Validar que la fila base exista y que tenga todas las columnas necesarias para inferencia.
2. Cargar los modelos y sus metadatos para el cultivo solicitado.
3. Construir la variante `base` como copia exacta de la fila original.
4. Construir cada escenario solicitado a partir de una copia de la fila base.
5. Aplicar únicamente las perturbaciones permitidas por el catálogo.
6. Reordenar columnas según el metadato del modelo antes de inferir.
7. Ejecutar regresor y clasificador para cada escenario.
8. Calcular deltas contra `base` en rendimiento esperado y probabilidad de riesgo alto.
9. Registrar qué columnas fueron tocadas y cuáles quedaron preservadas.
10. Devolver una estructura ordenada y, si el caller lo pide, persistirla como Parquet.

## Interfaz o contrato

- Función de dominio sugerida: `simulate_scenarios(baseline_row, scenarios) -> DataFrame`
- Endpoint asociado: `POST /escenario`
- Salida esperada: JSON o DataFrame con una fila por escenario solicitado, incluyendo la línea base

### Esquema mínimo de salida

| Campo                    | Tipo        | Descripción                                  |
| ------------------------ | ----------- | -------------------------------------------- |
| `codigo_dane`            | `str`       | Llave del municipio                          |
| `cultivo`                | `str`       | Cultivo evaluado                             |
| `año`                    | `int`       | Año de referencia de la simulación           |
| `escenario`              | `str`       | `base`, `seco`, `lluvioso` o `fertilizantes` |
| `rendimiento_esperado`   | `float`     | Salida del regresor                          |
| `prob_riesgo_alto`       | `float`     | Salida del clasificador                      |
| `etiqueta_riesgo`        | `str`       | Etiqueta final del riesgo                    |
| `delta_rendimiento_abs`  | `float`     | Diferencia contra la línea base              |
| `delta_rendimiento_pct`  | `float`     | Cambio relativo contra la línea base         |
| `delta_prob_riesgo_alto` | `float`     | Diferencia contra la línea base              |
| `features_modificadas`   | `list[str]` | Columnas tocadas por el shock                |
| `supuesto`               | `str`       | Texto corto del escenario aplicado           |

## Manejo de errores

- Debe fallar con `FileNotFoundError` si falta algún modelo o metadato.
- Debe fallar con `KeyError` si la fila base no tiene columnas requeridas.
- Debe fallar con `ValueError` si se solicita un escenario fuera del catálogo.
- Debe fallar con `ValueError` si el orden de features del modelo no puede reconstruirse.
- Debe registrar warning si una variable perturbada sale del rango físico esperado y necesita clamp.

## Salida esperada

- Una comparación determinista base vs. escenarios, lista para ser consumida por la API y el frontend.
- Una fila base inmutable y una fila por cada escenario solicitado.
- Trazabilidad explícita de qué features cambiaron y qué supuestos se aplicaron.

## Caso feliz

- Dada una fila base válida de un municipio y cultivo del MVP.
- Cuando se solicitan los escenarios `base`, `seco`, `lluvioso` y `fertilizantes`.
- Entonces el motor devuelve cuatro registros ordenados, con deltas calculados y sin modificar la fila original.

## Validación

- Dado un catálogo con un escenario no reconocido.
- Cuando se ejecuta la simulación.
- Entonces la función rechaza la entrada con un error claro y accionable.

- Dada una fila base válida.
- Cuando se ejecuta la simulación dos veces con los mismos parámetros.
- Entonces la salida debe ser idéntica en ambas ejecuciones.

## Validación independiente (validate_m5.py)

El script `specs/M5_simulacion_escenarios/validate_m5.py` debe:

1. Verificar que existen los modelos y metadatos necesarios para el cultivo solicitado.
2. Confirmar que `features_modificadas` solo contiene columnas permitidas por escenario.
3. Validar que los deltas se calculan siempre contra el escenario `base`.
4. Reportar `Exit Code 0` si todo es consistente; `1` si falla.

## Criterios de aceptación

- Claridad: el contrato distingue línea base, escenarios y deltas.
- Atomicidad: el módulo resuelve una sola responsabilidad de negocio.
- Trazabilidad: cada escenario declara qué features toca.
- Verificabilidad: la salida puede contrastarse contra una fila base fija.

## Restricciones técnicas

- No reentrenar modelos.
- No introducir aleatoriedad.
- No usar `produccion` ni `area_cosechada` del mismo año como features.
- No modificar la fila base; siempre trabajar sobre copias.
- Conservar el orden exacto de features definido por los metadatos de M2 y M3.
- Limitar los escenarios a transformaciones locales y deterministas.

## Dependencias

- Requiere: `modules/predictive/feature_builder.py`, `models/xgb_regressor_*.pkl`, `models/xgb_classifier_*.pkl`, sus metadatos y `shared/config.py`.
- Produce para: `POST /escenario` y la comparación visual de escenarios en el frontend.
