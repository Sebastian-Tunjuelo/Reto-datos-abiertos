# A3.2 - Asistente conversacional del MVP

## Metadatos

| Campo         | Valor                                                                                                                                                                                |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| ID            | A3.2                                                                                                                                                                                 |
| Módulo o área | `orchestrator/main.py` → `POST /chat`                                                                                                                                                |
| Depende de    | `docs/dominio/glosario_agricola.md`, `data/predicciones_con_explicacion.parquet`, M4, `modules/conversational/*`, `modules/explainability/*`, `data/eva_completa.parquet`, shared/\* |
| Produce para  | F4                                                                                                                                                                                   |
| Estado        | 📝 Borrador                                                                                                                                                                          |

## Agente

- role: Ingeniero backend / LLM-RAG.
- goal: Responder preguntas en lenguaje natural usando contexto recuperado de las predicciones, la explicabilidad y el glosario del dominio.
- backstory: El asistente debe sonar útil para UMATAs y productores, no como un chatbot genérico sin contexto agrícola.

## Objetivo

- Convertir una pregunta en lenguaje natural en una respuesta breve, clara y trazable, usando el contexto del modelo, SHAP, el glosario del dominio y los artefactos de explicación ya generados.

## Alcance

### Incluye

- `POST /chat` como endpoint principal de la API.
- Aceptar una pregunta libre y contexto opcional de municipio, cultivo, año o escenario.
- Recuperar información de predicciones, narrativas SHAP y glosario agrícola.
- Generar respuestas en tono campesino o institucional, según el modo solicitado.
- Exponer qué fuentes o artefactos se usaron para construir la respuesta.
- Señalar cuándo falta contexto local y hace falta precisar municipio/cultivo.

### No incluye

- Navegación web ni búsqueda en Internet.
- Entrenamiento o ajuste fino del modelo de lenguaje.
- Responder fuera del dominio agrícola del proyecto sin guardrails.
- Inventar datos meteorológicos, productivos o económicos.
- Persistencia de conversaciones como fuente primaria del sistema.

## Entradas

- Request JSON con una pregunta y, opcionalmente, contexto agrícola.
- `data/predicciones_con_explicacion.parquet` para recuperar predicciones y narrativas.
- `docs/dominio/glosario_agricola.md` como vocabulario controlado.
- M4 y artefactos de explicabilidad para factores y narrativas.
- `modules/conversational/rag.py` y `modules/conversational/prompts.py` para recuperación y prompting.

## Inputs esperados

| Campo       | Tipo  | Regla                                                                      |
| ----------- | ----- | -------------------------------------------------------------------------- |
| `pregunta`  | `str` | Requerida. No puede venir vacía.                                           |
| `municipio` | `str` | Opcional. Nombre display o código DANE del MVP.                            |
| `cultivo`   | `str` | Opcional. Debe resolverse a `Café`, `Cacao` o `Maíz` si se envía.          |
| `año`       | `int` | Opcional. Año de referencia de la pregunta.                                |
| `escenario` | `str` | Opcional. Útil si la pregunta se refiere a una simulación específica.      |
| `tono`      | `str` | Opcional. `campesino` o `institucional`. Si no se envía, usar `campesino`. |

## Lógica / Proceso sugerido

1. Normalizar el municipio y el cultivo cuando se suministren.
2. Validar que la pregunta no esté vacía.
3. Identificar si la consulta pide explicación de una predicción, comparación de escenarios, interpretación de SHAP o una respuesta general del dominio.
4. Recuperar contexto desde `predicciones_con_explicacion.parquet`, narrativas SHAP, series históricas y glosario.
5. Si se suministra municipio/cultivo/año, restringir la recuperación a ese contexto.
6. Si no existe contexto específico, responder con una explicación general y pedir precisión adicional en lugar de fabricar datos locales.
7. Construir el prompt con las plantillas del módulo conversacional y el tono solicitado.
8. Generar la respuesta con guardrails de dominio.
9. Devolver la respuesta junto con la lista de fuentes usadas y el contexto recuperado.
10. Si la consulta menciona que se necesita un reporte, indicar que existe el endpoint de reporte o marcarlo como disponible.

## Interfaz o contrato

### `POST /chat`

Request `ChatRequest`:

| Campo       | Tipo  | Nullable | Regla                                                                |
| ----------- | ----- | -------- | -------------------------------------------------------------------- |
| `pregunta`  | `str` | No       | Pregunta libre del usuario.                                          |
| `municipio` | `str` | Sí       | Nombre display o código DANE del MVP.                                |
| `cultivo`   | `str` | Sí       | `Café` \| `Cacao` \| `Maíz`.                                         |
| `año`       | `int` | Sí       | Año de referencia del contexto.                                      |
| `escenario` | `str` | Sí       | `base`, `seco`, `lluvioso` o `fertilizantes` si la pregunta lo pide. |
| `tono`      | `str` | Sí       | `campesino` \| `institucional`.                                      |

Response `ChatResponse`:

| Campo                | Tipo        | Nullable | Regla                                                                 |
| -------------------- | ----------- | -------- | --------------------------------------------------------------------- |
| `respuesta`          | `str`       | No       | Texto generado con lenguaje claro y trazable.                         |
| `tono_aplicado`      | `str`       | No       | Tono final usado por el asistente.                                    |
| `contexto_usado`     | `dict`      | No       | Resumen del municipio/cultivo/año/escenario efectivamente recuperado. |
| `fuentes`            | `list[str]` | No       | Artefactos o corpus usados para generar la respuesta.                 |
| `reporte_disponible` | `bool`      | No       | Indica si el usuario puede pasar al endpoint de reporte.              |

Ejemplo de respuesta:

```json
{
  "respuesta": "El riesgo es alto porque la lluvia estuvo por debajo de la media y el costo de insumos subió...",
  "tono_aplicado": "campesino",
  "contexto_usado": {
    "municipio": "Chaparral",
    "cultivo": "Café",
    "año": 2025,
    "escenario": "base"
  },
  "fuentes": [
    "predicciones_con_explicacion.parquet",
    "narrativa_riesgo",
    "glosario_agricola"
  ],
  "reporte_disponible": true
}
```

## Manejo de errores

| Situación                     | HTTP | Mensaje sugerido                                                              |
| ----------------------------- | ---- | ----------------------------------------------------------------------------- |
| Pregunta vacía                | 422  | `La pregunta no puede estar vacía`                                            |
| Municipio no reconocible      | 404  | `Municipio no encontrado en el MVP`                                           |
| Cultivo inválido              | 422  | `Cultivo inválido. Valores permitidos: Café, Cacao, Maíz`                     |
| Año inválido                  | 422  | `Año de contexto inválido`                                                    |
| Escenario inválido            | 422  | `Escenario inválido. Valores permitidos: base, seco, lluvioso, fertilizantes` |
| Corpus o índice no disponible | 503  | `Contexto conversacional no disponible`                                       |
| Error del proveedor LLM/RAG   | 502  | `Error al generar la respuesta conversacional`                                |

## Salida esperada

- Una respuesta clara, breve y accionable.
- Lenguaje adaptado al tono solicitado.
- Trazabilidad mínima del contexto usado.
- Sin inventar datos cuando no existe contexto local suficiente.

## Caso feliz

- Dada la pregunta `Explícame este riesgo en lenguaje sencillo para un productor`.
- Cuando el frontend invoca `POST /chat` con contexto de municipio y cultivo.
- Entonces el asistente responde con una explicación breve, en tono adecuado, y señala las fuentes utilizadas.

## Validación

- Dada una pregunta vacía.
- Cuando se invoca el endpoint.
- Entonces la API responde con 422.
- Dado un municipio y cultivo válidos.
- Cuando se invoca el endpoint.
- Entonces la respuesta contiene texto no vacío, tono aplicado y contexto recuperado.
- Dado contexto insuficiente.
- Cuando se invoca el endpoint.
- Entonces la API responde con una aclaración útil sin inventar datos locales.

## Criterios de aceptación

- La respuesta está grounded en contexto recuperado y no en invenciones del modelo.
- El tono se adapta al usuario objetivo.
- La salida permite auditar qué fuentes se consultaron.
- El asistente puede operar con o sin contexto local, pero sin falsificar precisión.

## Restricciones técnicas

- No usar navegación web ni fuentes externas no controladas.
- No usar el LLM como reemplazo de los datos del proyecto.
- No devolver afirmaciones específicas sin sustento recuperado.
- No acoplar el chat al entrenamiento de modelos.
