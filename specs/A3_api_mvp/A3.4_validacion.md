## Agente

**Rol**: Eres un ingeniero de QA y MLOps encargado de certificar que la API expone el contrato esperado y que el comportamiento observable coincide con las specs de A3.

**Mentalidad**: Piensa como un auditor externo. No verificas la implementación interna; verificas el contrato HTTP, los esquemas de respuesta, los errores observables y la trazabilidad mínima.

**Restricciones**:

- No llamar funciones privadas de `modules/` para validar el contrato
- No modificar archivos de datos ni modelos
- Cada aserción debe tener un mensaje descriptivo con prefijo `❌`
- Si una aserción falla, el script debe detenerse inmediatamente con `AssertionError`
- El script debe poder ejecutarse de forma aislada

---

## Objetivo

Verificar que los tres endpoints del bloque A3 funcionan como contrato público:

1. `POST /escenario`
2. `POST /chat`
3. `GET /reporte/{municipio}/{cultivo}`

Además, comprobar que el contrato de error, la normalización de entradas y la trazabilidad mínima se mantienen consistentes.

## Checklist de criterios de aceptación

### Disponibilidad básica

- [x] `orchestrator.main.app` es importable
- [x] `GET /openapi.json` responde y contiene los 3 endpoints de A3

### `POST /escenario`

- [x] Responde `200 OK` para una entrada válida
- [x] Devuelve `codigo_dane`, `municipio`, `departamento`, `cultivo`, `año`, `escenarios_solicitados` y `resultados`
- [x] `escenarios_solicitados` incluye `base` en primer lugar
- [x] `resultados` devuelve una fila por escenario
- [x] `base` tiene deltas en cero respecto de sí mismo
- [x] Todos los escenarios devueltos pertenecen al catálogo canónico
- [x] Un escenario inválido responde `422`
- [x] Un municipio inválido responde `404`
- [x] Un cultivo inválido responde `422`

### `POST /chat`

- [x] Responde `200 OK` para una pregunta válida
- [x] Devuelve `respuesta`, `tono_aplicado`, `contexto_usado`, `fuentes` y `reporte_disponible`
- [x] `respuesta` no está vacía
- [x] `contexto_usado` refleja la entrada cuando se suministra municipio/cultivo/año
- [x] Una pregunta vacía responde `422`
- [x] Un cultivo inválido responde `422`
- [x] Un municipio inválido responde `404`

### `GET /reporte/{municipio}/{cultivo}`

- [x] Para una combinación válida responde `200 OK`
- [x] En modo PDF devuelve `Content-Type: application/pdf`
- [x] En modo texto devuelve un JSON estable con `contenido_texto` y `fuentes`
- [x] El reporte hace referencia a una predicción explicada existente
- [x] Un formato inválido responde `422`
- [x] Un municipio inválido responde `404`
- [x] Un cultivo inválido responde `422`

---

## Validación funcional mínima esperada

- `POST /escenario` debe conservar la fila base y calcular deltas contra `base`.
- `POST /chat` debe devolver una respuesta grounded y no vacía.
- `GET /reporte/{municipio}/{cultivo}` debe producir un artefacto descargable o texto estructurado con fuentes rastreables.

---

## Resultado esperado del script

- `Exit Code 0` si todas las aserciones pasan.
- `AssertionError` en el primer fallo con un mensaje claro y accionable.
- No modificar ningún artefacto persistente del proyecto.

---

## Criterios de aceptación

- La validación cubre OpenAPI, happy path y errores relevantes.
- Las respuestas observables coinciden con las specs A3.1, A3.2 y A3.3.
- El script puede ejecutarse de forma independiente y repetible.
