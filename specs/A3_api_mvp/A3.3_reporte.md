# A3.3 - Generación de reporte por municipio y cultivo

## Metadatos

| Campo         | Valor                                                                                                                                                                                      |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| ID            | A3.3                                                                                                                                                                                       |
| Módulo o área | `orchestrator/main.py` → `GET /reporte/{municipio}/{cultivo}`                                                                                                                              |
| Depende de    | A2, A3.1, A3.2, `data/predicciones_con_explicacion.parquet`, `data/eva_completa.parquet`, `data/clima_agregado.parquet`, `modules/conversational/*`, `modules/explainability/*`, shared/\* |
| Produce para  | F2, F4                                                                                                                                                                                     |
| Estado        | ✅ Completo                                                                                                                                                                                |

## Agente

- role: Ingeniero backend / generación de documentos.
- goal: Construir un reporte compartible que sintetice contexto histórico, predicción, factores SHAP, escenarios y recomendación.
- backstory: El reporte debe servir para UMATAs, extensionistas y productores, con un formato descargable y legible fuera de la plataforma.

## Objetivo

- Generar un reporte descargable para un municipio y cultivo del MVP usando la última predicción explicada disponible y el contexto histórico del proyecto.

## Alcance

### Incluye

- `GET /reporte/{municipio}/{cultivo}` como endpoint read-only.
- Aceptar municipio por nombre display o código DANE.
- Aceptar cultivo del MVP.
- Resolver la última predicción explicada disponible para el par municipio + cultivo.
- Enriquecer el reporte con serie histórica, clima agregado y factores SHAP.
- Permitir salida en PDF por defecto y salida textual bajo formato explícito.
- Devolver un artefacto listo para descargar o compartir.

### No incluye

- Reentrenamiento de modelos.
- Descarga de datos crudos.
- Escritura de resultados en PostgreSQL como parte del contrato.
- Generación de reportes sin contexto mínimo de predicción o explicación.
- Conversión automática a otros formatos fuera de PDF o texto.

## Entradas

- `municipio` y `cultivo` en la ruta.
- Contexto persistido en `data/predicciones_con_explicacion.parquet`.
- Serie histórica de EVA y clima agregado para complementar el documento.
- Narrativas y top features ya generadas por M4.
- Plantillas y lógica del módulo conversacional de reportes.

## Inputs esperados

| Campo       | Tipo  | Regla                                               |
| ----------- | ----- | --------------------------------------------------- |
| `municipio` | `str` | Nombre display o código DANE del MVP.               |
| `cultivo`   | `str` | `Café`, `Cacao` o `Maíz`.                           |
| `formato`   | `str` | Opcional. `pdf` o `texto`. Si se omite, usar `pdf`. |

## Lógica / Proceso sugerido

1. Normalizar municipio y cultivo.
2. Resolver el municipio a código DANE canónico y validar que pertenezca al MVP.
3. Validar que el cultivo pertenezca al MVP.
4. Localizar la última predicción explicada disponible para el par municipio + cultivo.
5. Cargar la serie histórica de rendimiento y la serie climática agregada.
6. Recuperar los factores SHAP, la narrativa de riesgo y, si existe, la comparación de escenarios.
7. Construir secciones de reporte con lenguaje claro para UMATA y productor.
8. Renderizar el documento en PDF o devolver el texto estructurado según el formato solicitado.
9. Si el formato es PDF, devolver headers de descarga coherentes.
10. Si el formato es texto, devolver un JSON estable con el contenido textual y metadatos.

## Interfaz o contrato

### `GET /reporte/{municipio}/{cultivo}`

#### Opción PDF por defecto

- Respuesta con `Content-Type: application/pdf`.
- El archivo debe ser descargable con un nombre estable que incluya municipio y cultivo.
- El contenido debe ser la versión consolidada del reporte.

#### Opción textual

Si se solicita `formato=texto`, la respuesta puede devolver JSON con el siguiente contrato mínimo:

| Campo             | Tipo        | Nullable | Regla                                        |
| ----------------- | ----------- | -------- | -------------------------------------------- |
| `codigo_dane`     | `str`       | No       | 5 dígitos.                                   |
| `municipio`       | `str`       | No       | Title Case.                                  |
| `departamento`    | `str`       | No       | Title Case.                                  |
| `cultivo`         | `str`       | No       | `Café` \| `Cacao` \| `Maíz`.                 |
| `año_referencia`  | `int`       | No       | Año de la predicción ancla.                  |
| `titulo`          | `str`       | No       | Título del reporte.                          |
| `contenido_texto` | `str`       | No       | Resumen completo en texto.                   |
| `secciones`       | `list[str]` | No       | Lista ordenada de secciones incluidas.       |
| `fuentes`         | `list[str]` | No       | Artefactos usados para construir el reporte. |

Ejemplo de contenido textual:

```json
{
  "codigo_dane": "73168",
  "municipio": "Chaparral",
  "departamento": "Tolima",
  "cultivo": "Café",
  "año_referencia": 2025,
  "titulo": "Reporte municipal de riesgo y recomendación - Chaparral / Café",
  "contenido_texto": "Resumen ejecutivo...",
  "secciones": [
    "Resumen ejecutivo",
    "Histórico",
    "Clima",
    "Riesgo",
    "Escenarios",
    "Recomendación"
  ],
  "fuentes": [
    "predicciones_con_explicacion.parquet",
    "eva_completa.parquet",
    "clima_agregado.parquet"
  ]
}
```

## Manejo de errores

| Situación                        | HTTP | Mensaje sugerido                                          |
| -------------------------------- | ---- | --------------------------------------------------------- |
| Municipio no reconocible         | 404  | `Municipio no encontrado en el MVP`                       |
| Cultivo inválido                 | 422  | `Cultivo inválido. Valores permitidos: Café, Cacao, Maíz` |
| Formato no soportado             | 422  | `Formato inválido. Valores permitidos: pdf, texto`        |
| No existe predicción explicada   | 404  | `No hay contexto suficiente para generar el reporte`      |
| Corpus o plantilla no disponible | 503  | `Reportes no disponibles`                                 |
| Error inesperado al generar      | 500  | `Error interno al generar el reporte`                     |

## Salida esperada

- Un reporte listo para descargar o compartir.
- Un resumen que combine predicción, histórico, SHAP y recomendación.
- Un formato consistente para UMATA o para impresión/WhatsApp.
- Una traza clara de las fuentes usadas.

## Caso feliz

- Dado `municipio = "Chaparral"` y `cultivo = "Café"`.
- Cuando el frontend invoca `GET /reporte/Chaparral/Café`.
- Entonces la API devuelve un PDF descargable o el texto estructurado con las secciones definidas.

## Validación

- Dado un municipio y cultivo válidos.
- Cuando se solicita el reporte en PDF.
- Entonces la respuesta tiene `Content-Type: application/pdf`.
- Dado un municipio y cultivo válidos.
- Cuando se solicita `formato=texto`.
- Entonces la respuesta contiene un JSON estable con `contenido_texto` y `fuentes`.
- Dado un formato inválido.
- Cuando se solicita el endpoint.
- Entonces la API responde con 422.

## Criterios de aceptación

- El reporte se basa en una predicción explicada existente.
- El documento es entendible para UMATAs y productores.
- La salida es reproducible para la misma entrada y el mismo contexto.
- No se inventan datos fuera del corpus del proyecto.

## Restricciones técnicas

- No usar navegación web.
- No recalcular el entrenamiento ni la explicación.
- No generar contenido sin contexto mínimo de predicción.
- No modificar los artefactos fuente durante la generación.
