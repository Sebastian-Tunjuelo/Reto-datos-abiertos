## Agente

- role: Ingeniero backend / ML integrado.
- goal: Convertir una petición municipio + cultivo + año en una comparación determinista entre línea base y escenarios.
- backstory: Este endpoint reutiliza la lógica de M5 para responder qué cambia si el año es seco, lluvioso o si sube el costo de insumos.

## Objetivo

- Devolver una simulación de escenarios para un municipio y cultivo del MVP, usando como base la última fila histórica disponible y manteniendo trazabilidad explícita de qué variables fueron perturbadas.

## Alcance

### Incluye

- `POST /escenario` como endpoint principal de la API.
- Aceptar municipio por nombre display o código DANE.
- Aceptar cultivo del MVP y año objetivo de planeación.
- Resolver la fila base histórica antes de simular.
- Permitir el catálogo canónico `base`, `seco`, `lluvioso` y `fertilizantes`.
- Incluir siempre la línea base como referencia en la respuesta.
- Ejecutar el motor determinista definido en `modules/predictive/scenarios.py`.
- Calcular deltas contra la línea base y devolver la trazabilidad de las features tocadas.

### No incluye

- Chat o RAG.
- Generación de reportes PDF/texto.
- Entrenamiento o recalibración de modelos.
- Búsqueda de escenarios fuera del catálogo canónico.
- Pronóstico climático real.
- Persistencia de resultados en PostgreSQL.

## Entradas

- Request JSON con `municipio`, `cultivo`, `año` y, opcionalmente, `escenarios`.
- `data/feature_matrix.parquet` para ubicar la fila base.
- Modelos y metadatos de M2 y M3 para el cultivo solicitado.
- `modules/predictive/scenarios.py` para aplicar shocks deterministas.

## Inputs esperados

| Campo        | Tipo        | Regla                                                                                          |
| ------------ | ----------- | ---------------------------------------------------------------------------------------------- |
| `municipio`  | `str`       | Nombre display o código DANE del MVP.                                                          |
| `cultivo`    | `str`       | Debe resolverse a `Café`, `Cacao` o `Maíz`.                                                    |
| `año`        | `int`       | Año objetivo de planeación; debe ser posterior al último año histórico disponible para el par. |
| `escenarios` | `list[str]` | Opcional. Subconjunto del catálogo canónico. Si se omite, usar el catálogo completo.           |

## Lógica / Proceso sugerido

1. Normalizar municipio y cultivo con las reglas canónicas del proyecto.
2. Resolver el municipio a código DANE y verificar que pertenezca al MVP.
3. Validar que el cultivo pertenezca a `CULTIVOS_MVP`.
4. Validar que el año objetivo sea un entero y represente una solicitud futura.
5. Cargar `data/feature_matrix.parquet` y ubicar la última fila histórica disponible para el par municipio + cultivo.
6. Verificar que existan las features mínimas necesarias para el motor de escenarios.
7. Cargar el regressor y classifier del cultivo junto con sus metadatos.
8. Normalizar la lista de escenarios solicitados, deduplicar conservando orden y anteponer `base` si no está presente.
9. Rechazar cualquier escenario fuera del catálogo canónico.
10. Invocar `simulate_scenarios()` y preservar el orden base primero, seguido por los escenarios solicitados.
11. Enriquecer la salida con los identificadores canónicos del municipio y el cultivo.
12. Devolver un JSON estable y reproducible.

## Interfaz o contrato

### `POST /escenario`

Request `EscenarioRequest`:

| Campo        | Tipo        | Nullable | Regla                                                                  |
| ------------ | ----------- | -------- | ---------------------------------------------------------------------- |
| `municipio`  | `str`       | No       | Nombre o código DANE.                                                  |
| `cultivo`    | `str`       | No       | `Café` \| `Cacao` \| `Maíz`.                                           |
| `año`        | `int`       | No       | Año objetivo de simulación.                                            |
| `escenarios` | `list[str]` | Sí       | `base`, `seco`, `lluvioso`, `fertilizantes`. `base` siempre se incluye |

Response `EscenarioResponse`:

| Campo                    | Tipo         | Nullable | Regla                                                                 |
| ------------------------ | ------------ | -------- | --------------------------------------------------------------------- |
| `codigo_dane`            | `str`        | No       | 5 dígitos.                                                            |
| `municipio`              | `str`        | No       | Title Case.                                                           |
| `departamento`           | `str`        | No       | Title Case.                                                           |
| `cultivo`                | `str`        | No       | `Café` \| `Cacao` \| `Maíz`.                                          |
| `año`                    | `int`        | No       | Echo del año solicitado.                                              |
| `escenarios_solicitados` | `list[str]`  | No       | Lista final enviada al motor, con `base` en primer lugar.             |
| `resultados`             | `list[dict]` | No       | Una fila por escenario, en el mismo orden de `escenarios_solicitados` |

Cada elemento de `resultados` debe seguir el contrato de M5.1:

| Campo                    | Tipo        | Nullable | Regla                               |
| ------------------------ | ----------- | -------- | ----------------------------------- |
| `escenario`              | `str`       | No       | Escenario aplicado.                 |
| `rendimiento_esperado`   | `float`     | No       | t/ha estimadas.                     |
| `prob_riesgo_alto`       | `float`     | No       | Fracción en rango `[0, 1]`.         |
| `etiqueta_riesgo`        | `str`       | No       | `Bajo`, `Medio` o `Alto`.           |
| `delta_rendimiento_abs`  | `float`     | No       | Diferencia contra `base`.           |
| `delta_rendimiento_pct`  | `float`     | No       | Cambio relativo contra `base`.      |
| `delta_prob_riesgo_alto` | `float`     | No       | Diferencia contra `base`.           |
| `features_modificadas`   | `list[str]` | No       | Columnas tocadas por el shock.      |
| `supuesto`               | `str`       | No       | Texto corto del escenario aplicado. |

Ejemplo de respuesta:

```json
{
  "codigo_dane": "73168",
  "municipio": "Chaparral",
  "departamento": "Tolima",
  "cultivo": "Café",
  "año": 2025,
  "escenarios_solicitados": ["base", "seco", "lluvioso"],
  "resultados": [
    {
      "escenario": "base",
      "rendimiento_esperado": 0.82,
      "prob_riesgo_alto": 0.71,
      "etiqueta_riesgo": "Alto",
      "delta_rendimiento_abs": 0.0,
      "delta_rendimiento_pct": 0.0,
      "delta_prob_riesgo_alto": 0.0,
      "features_modificadas": [],
      "supuesto": "Línea base original"
    }
  ]
}
```

## Manejo de errores

| Situación                           | HTTP | Mensaje sugerido                                                              |
| ----------------------------------- | ---- | ----------------------------------------------------------------------------- |
| Municipio no reconocible            | 404  | `Municipio no encontrado en el MVP`                                           |
| Cultivo inválido                    | 422  | `Cultivo inválido. Valores permitidos: Café, Cacao, Maíz`                     |
| Año no entero o no futuro           | 422  | `Año objetivo inválido para simulación`                                       |
| Escenario fuera del catálogo        | 422  | `Escenario inválido. Valores permitidos: base, seco, lluvioso, fertilizantes` |
| No existe una fila base histórica   | 404  | `No hay datos históricos suficientes para la combinación solicitada`          |
| Falta un modelo o metadato          | 503  | `Artefactos de escenarios no disponibles`                                     |
| La tabla maestra no está disponible | 503  | `feature_matrix.parquet no disponible`                                        |
| Error inesperado de inferencia      | 500  | `Error interno en la API de escenarios`                                       |

## Salida esperada

- Una comparación determinista base vs. escenarios.
- Una fila base inmutable y una fila por cada escenario solicitado.
- Trazabilidad explícita de qué features cambiaron en cada shock.
- Deltas calculados siempre contra la línea base.

## Caso feliz

- Dado `municipio = "Chaparral"`, `cultivo = "Café"`, `año = 2025` y escenarios `base`, `seco` y `lluvioso`.
- Cuando el frontend invoca `POST /escenario`.
- Entonces la API devuelve la comparación completa con `base` primero y los deltas correctamente calculados.

## Validación

- Dado un escenario no reconocido.
- Cuando se invoca el endpoint.
- Entonces la API responde con 422.
- Dada una fila base válida.
- Cuando se invoca el endpoint dos veces con los mismos parámetros.
- Entonces la salida debe ser idéntica en ambas ejecuciones.
- Dado un municipio o cultivo inválido.
- Cuando se invoca el endpoint.
- Entonces la API responde con 404 o 422 según corresponda.

## Criterios de aceptación

- [x] El escenario base siempre se devuelve y se usa como referencia de todos los deltas.
- [x] La respuesta preserva el orden solicitado de escenarios.
- [x] El motor no muta la fila base.
- [x] No se aceptan escenarios fuera del catálogo canónico.
- [x] La salida es reproducible y auditable.

## Restricciones técnicas

- No reentrenar modelos.
- No introducir aleatoriedad.
- No usar `produccion` ni `area_cosechada` del mismo año como features.
- No modificar la fila base; siempre trabajar sobre copias.
- Conservar el orden exacto de features definido por los metadatos de M2 y M3.
- Limitar los escenarios a transformaciones locales y deterministas.
