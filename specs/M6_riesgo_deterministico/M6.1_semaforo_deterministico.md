- Produce para: `POST /predecir`, `POST /escenario`
- Estado: ✅ Completo

## Contexto del problema

El sistema actual tiene dos defectos críticos que hacen que las predicciones sean inútiles en producción:

### Defecto 1 — El clasificador de riesgo no generaliza

El `XGBClassifier` entrenado en M3 produce AUC de 0.28 en test para Café (peor que azar) y AUC de 0.54 para Cacao. Los datos de entrenamiento son insuficientes (28–169 filas por cultivo) para que un modelo de ML aprenda patrones de riesgo. El resultado es que casi todos los municipios reciben `prob_riesgo_alto` alta y la etiqueta "Alto" sin importar el contexto real.

### Defecto 2 — La predicción no varía entre años

En `POST /predecir`, el endpoint toma la última fila histórica de `feature_matrix.parquet` y solo cambia el campo `año`. Todas las features (rendimiento_t1, rendimiento_prom3a, clima, etc.) quedan congeladas en el último año histórico. El modelo recibe exactamente el mismo vector de entrada para 2025, 2026 o 2027 → produce exactamente el mismo resultado.

---

## Objetivo

1. Reemplazar el `XGBClassifier` por una función determinística que asigne el semáforo de riesgo (Bajo/Medio/Alto) comparando el rendimiento predicho por el regresor contra el promedio histórico del municipio y cultivo.
2. Hacer que la predicción por año sea dinámica: actualizar `rendimiento_t1`, `rendimiento_prom3a` y `tendencia_rend_3a` usando el rendimiento predicho del año anterior como proxy cuando se pide un año futuro.

## Lógica / Proceso

### Parte A — Función determinística de semáforo (`risk_rules.py`)

La lógica de riesgo deja de depender de un clasificador ML y pasa a ser una regla explícita basada en la caída de rendimiento:

```
caida_pct = (promedio_historico - rend_predicho) / promedio_historico

si caida_pct >= UMBRAL_RIESGO_ALTO  → etiqueta = "Alto",  prob_riesgo = caida_pct
si caida_pct >= UMBRAL_RIESGO_MEDIO → etiqueta = "Medio", prob_riesgo = caida_pct
si caida_pct < UMBRAL_RIESGO_MEDIO  → etiqueta = "Bajo",  prob_riesgo = caida_pct
si rend_predicho >= promedio_historico → etiqueta = "Bajo", prob_riesgo = 0.0
```

`prob_riesgo` se expone como la caída porcentual normalizada (0.0–1.0), no como probabilidad ML. Esto mantiene compatibilidad con el campo `prob_riesgo_alto` del response sin cambiar el contrato de la API.

**Umbrales revisados en `shared/config.py`:**

```python
UMBRAL_RIESGO_ALTO  = 0.15  # caída > 15% vs promedio histórico → Alto
UMBRAL_RIESGO_MEDIO = 0.07  # caída > 7%  vs promedio histórico → Medio
```

Estos valores son coherentes con la definición original de `target_riesgo` (caída > 15%) y dan una distribución más equilibrada entre las tres categorías.

### Parte B — Predicción dinámica por año (`POST /predecir`)

El endpoint actual congela las features en el último año histórico. La corrección es:

1. Cargar la serie histórica completa del municipio+cultivo desde `feature_matrix.parquet`
2. Calcular el `promedio_historico` de rendimiento usando todos los años históricos disponibles (sin incluir el año objetivo)
3. Para el año objetivo `N`:
   - Si `N == max_año_historico + 1`: usar `last_row` como base (comportamiento actual, correcto)
   - Si `N > max_año_historico + 1`: proyectar iterativamente:
     - Predecir rendimiento para `max_año_historico + 1` usando el regresor
     - Usar ese rendimiento predicho como `rendimiento_t1` para el siguiente año
     - Actualizar `rendimiento_prom3a` con la ventana deslizante de los últimos 3 valores conocidos/predichos
     - Actualizar `tendencia_rend_3a` con la pendiente de los últimos 3 valores
     - Repetir hasta llegar al año `N`
4. Calcular la etiqueta de riesgo con `calcular_etiqueta_riesgo(rend_predicho, promedio_historico)`

**Features que se actualizan dinámicamente:**
- `rendimiento_t1` ← rendimiento predicho del año anterior
- `rendimiento_prom3a` ← promedio de los últimos 3 rendimientos (históricos + predichos)
- `tendencia_rend_3a` ← pendiente lineal de los últimos 3 rendimientos
- `año` ← año objetivo

**Features que permanecen constantes** (no cambian entre años futuros):
- Clima (`prec_acum_mm`, `temp_media_c`, etc.) — sin datos futuros disponibles
- Aptitud territorial (`pct_alta`, `pct_media`, etc.) — estática por definición
- Agroinsumos (`indice_agroinsumos`, `percentil_fertilizantes`) — último valor conocido
- `area_sembrada_t1` — último valor conocido

### Parte C — Escenarios (`POST /escenario`)

El endpoint de escenarios ya modifica features climáticas y de agroinsumos sobre una fila base. Solo necesita:
1. Usar `calcular_etiqueta_riesgo` en lugar del clasificador para asignar la etiqueta
2. Calcular `promedio_historico` de la misma forma que en `/predecir`

No se cambia la lógica de modificación de features de los escenarios (seco, lluvioso, fertilizantes).

## Casos de prueba

### Caso 1 — Riesgo Alto
- Dado: `rend_predicho=0.8`, `promedio_historico=1.2`
- Caída: `(1.2 - 0.8) / 1.2 = 0.333` → 33.3%
- Entonces: `etiqueta="Alto"`, `prob_riesgo=0.333`

### Caso 2 — Riesgo Medio
- Dado: `rend_predicho=1.1`, `promedio_historico=1.2`
- Caída: `(1.2 - 1.1) / 1.2 = 0.083` → 8.3%
- Entonces: `etiqueta="Medio"`, `prob_riesgo=0.083`

### Caso 3 — Riesgo Bajo (sin caída)
- Dado: `rend_predicho=1.3`, `promedio_historico=1.2`
- Caída: negativa → 0.0
- Entonces: `etiqueta="Bajo"`, `prob_riesgo=0.0`

### Caso 4 — Predicción dinámica multi-año
- Dado: último año histórico = 2024, se pide año = 2026
- Entonces: el modelo predice 2025 primero, usa ese resultado como `rendimiento_t1` para predecir 2026
- Los resultados de 2025 y 2026 deben ser distintos entre sí

### Caso 5 — Año demasiado lejano
- Dado: último año histórico = 2024, se pide año = 2031
- Entonces: HTTP 422 con mensaje claro

### Caso 6 — promedio_historico = 0
- Dado: municipio sin historial de rendimiento
- Entonces: `etiqueta="Bajo"`, `prob_riesgo=0.0`, warning en log
