# A1.1 - Catálogo de municipios del MVP

## Metadatos

| Campo         | Valor                                             |
| ------------- | ------------------------------------------------- |
| ID            | A1.1                                              |
| Módulo o área | `orchestrator/main.py` → `GET /municipios`        |
| Depende de    | `shared/dane_codes.py`, `shared/normalization.py` |
| Produce para  | A1.2, A1.3, F1, F2, F3                            |
| Estado        | ✅ Completado                                     |

## Agente

- role: Ingeniero backend FastAPI.
- goal: Exponer el catálogo canónico de municipios del MVP como una lista estable, sin filtros ni lógica adicional.
- backstory: Esta capa es la fuente de verdad para poblar selectores y evitar nombres hardcodeados en el frontend.

## Objetivo

- Devolver siempre los 15 municipios del MVP en el mismo orden canónico del proyecto.

## Alcance

### Incluye

- `GET /municipios` como endpoint read-only.
- Formateo de cada municipio con código DANE, nombre display y departamento.
- Orden estable según `MVP_CODIGOS`.

### No incluye

- Búsqueda por texto.
- Filtros por departamento.
- Paginación.
- Persistencia en base de datos.

## Entradas

- `MVP_CODIGOS`
- `DANE_TO_NAME`
- `DANE_TO_DEPT`

## Inputs esperados

- Ninguno. El endpoint no recibe body, path params ni query params.

## Lógica / Proceso sugerido

1. Cargar el catálogo desde `shared/dane_codes.py` al iniciar la app.
2. Recorrer `MVP_CODIGOS` en el orden definido por el proyecto.
3. Construir la salida con `codigo_dane`, `municipio` y `departamento`.
4. Verificar que no existan duplicados ni códigos sin resolución en los mapas.
5. Si el catálogo está inconsistente, fallar rápido con un error interno en vez de devolver una lista parcial.

## Interfaz o contrato

### `GET /municipios`

Respuesta esperada:

```json
[
  {
    "codigo_dane": "73001",
    "municipio": "Ibagué",
    "departamento": "Tolima"
  }
]
```

Esquema `MunicipioItem`:

| Campo          | Tipo  | Nullable | Regla       |
| -------------- | ----- | -------- | ----------- |
| `codigo_dane`  | `str` | No       | 5 dígitos.  |
| `municipio`    | `str` | No       | Title Case. |
| `departamento` | `str` | No       | Title Case. |

## Manejo de errores

| Situación                         | HTTP | Mensaje sugerido                       |
| --------------------------------- | ---- | -------------------------------------- |
| Catálogo inconsistente en memoria | 500  | `Catálogo de municipios inconsistente` |

## Salida esperada

- Lista de 15 municipios únicos.
- Orden estable.
- Nombres y departamentos normalizados para frontend.

## Caso feliz

- Dado que el servicio está levantado.
- Cuando el frontend llama `GET /municipios`.
- Entonces recibe exactamente 15 registros del MVP en el orden canónico.

## Validación

- Dado un catálogo completo en `shared/dane_codes.py`.
- Cuando se ejecuta el endpoint.
- Entonces la respuesta contiene 15 registros, sin duplicados y con códigos de 5 dígitos.

## Criterios de aceptación

- `shared/dane_codes.py` es la fuente única de verdad.
- El orden no cambia entre ejecuciones.
- No se aceptan nombres inventados ni variantes ortográficas fuera del catálogo.

## Restricciones técnicas

- No usar base de datos.
- No usar consultas externas.
- No introducir paginación ni búsqueda parcial.
