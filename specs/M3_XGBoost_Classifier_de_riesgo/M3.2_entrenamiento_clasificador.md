## Agente

**Rol**: Eres un ingeniero de ML responsable del entrenamiento de clasificadores binarios por cultivo.

**Mentalidad**: Prioriza split temporal, reproducibilidad y trazabilidad de artefactos. Nunca uses test para ajustar hiperparámetros.

**Restricciones**:

- No entrenar un modelo global
- No usar split aleatorio
- No usar data de test en training ni en la búsqueda de hiperparámetros
- No usar columnas de fuga como features

---

## Objetivo

Entrenar un `XGBClassifier` por cultivo usando split temporal estricto y mitigación de desbalance para predecir `target_riesgo`.

---

## Inputs

- DataFrame validado y enriquecido con `target_riesgo` proveniente de M3.1
- Constantes de `shared/config.py`: `CULTIVOS_MVP`, `TRAIN_HASTA`, `VAL_AÑO`, `TEST_DESDE`, `MODELS_DIR`

---

## Inputs esperados

| Columna                  | Requerida | Regla                                                                 |
| ------------------------ | --------- | --------------------------------------------------------------------- |
| `target_riesgo`          | Sí        | Binaria, `0`/`1`; los `NaN` se excluyen                               |
| `año`                    | Sí        | Se usa solo para el split temporal                                    |
| `cultivo`                | Sí        | Debe pertenecer a `CULTIVOS_MVP`                                      |
| `señal_riesgo_economico` | Sí        | Se codifica de forma case-insensitive con orden `bajo < medio < alto` |
| Features numéricas de M1 | Sí        | Se conservan siempre que no sean columnas de fuga                     |

Columnas prohibidas como features: `codigo_dane`, `municipio`, `departamento`, `año`, `target_rendimiento`, `target_riesgo`, `promedio_historico`, `umbral_riesgo`.

---

## Parámetros base del modelo

```python
XGBClassifier(
   objective="binary:logistic",
   n_estimators=500,
   learning_rate=0.05,
   max_depth=6,
   subsample=0.8,
   colsample_bytree=0.8,
   reg_alpha=0.0,
   reg_lambda=1.0,
   random_state=42,
   n_jobs=-1,
   eval_metric="auc",
)
```

Entrenar con `early_stopping_rounds=30` usando el split de validación.

---

## Lógica de entrenamiento

```text
Para cada cultivo en CULTIVOS_MVP:
1. Filtrar df por cultivo
2. Eliminar filas con target_riesgo nulo
3. Construir split temporal:
   - train: año <= TRAIN_HASTA
   - val:   año == VAL_AÑO
   - test:  año >= TEST_DESDE
4. Verificar que train/val/test tengan filas y ambos valores de clase
5. Seleccionar features y codificar señal_riesgo_economico de forma case-insensitive con orden `bajo < medio < alto`
6. Calcular scale_pos_weight = negativos / positivos sobre train
7. Entrenar XGBClassifier con early stopping sobre val
8. Conservar modelo + metadata para M3.3
```

def train_classifiers(

   ...

El retorno debe incluir, por cultivo, el modelo entrenado, la lista final de features, el `scale_pos_weight`, las dimensiones de los splits y el `best_iteration`.

---

## Manejo de errores

| Situación                                              | Comportamiento                                                         |
| ------------------------------------------------------ | ---------------------------------------------------------------------- |
| Split train/val/test vacío                             | `ValueError` con el cultivo afectado                                   |
| Un split contiene una sola clase                       | `ValueError` porque las métricas de clasificación dejan de ser válidas |
| No quedan features utilizables                         | `ValueError`                                                           |
| `target_riesgo` no existe o es nulo en todas las filas | `ValueError`                                                           |

---

## Salida esperada

Tres modelos `XGBClassifier` entrenados en memoria, uno por cultivo, listos para evaluarse y persistirse en M3.3.

---

## Caso feliz

- Dado un dataframe con `target_riesgo` y features válidas
- Cuando se entrena por cultivo con split temporal
- Entonces se obtienen tres clasificadores reproducibles y comparables

---

## Validación

- Dado el split de train de cada cultivo
- Cuando se calcula el desbalance
- Entonces `scale_pos_weight` coincide con `negativos / positivos`
- Dado el split de validación
- Cuando se activa early stopping
- Entonces el mejor booster se selecciona sin usar test

---

## Criterios de aceptación

- [ ] Se entrena un modelo por cultivo en `CULTIVOS_MVP`
- [ ] Se usa split temporal exacto (train/val/test por año)
- [ ] No se usan columnas de fuga como features
- [ ] Se calcula `scale_pos_weight` desde el train de cada cultivo
- [ ] `señal_riesgo_economico` se codifica de forma case-insensitive con orden `bajo < medio < alto`
- [ ] Las filas sin `target_riesgo` calculable se excluyen del entrenamiento

---

## Restricciones técnicas

- Emplear la API Scikit-Learn de XGBoost (`xgboost.XGBClassifier`)
- Mantener `NaN` nativo en features donde aplique
- No usar test para seleccionar hiperparámetros
