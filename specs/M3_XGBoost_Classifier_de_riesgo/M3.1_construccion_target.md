---

## Objetivo

Construir la etiqueta binaria `target_riesgo` por fila usando una caída de rendimiento mayor al 15% frente al promedio histórico calculado solo con años anteriores del mismo `codigo_dane` y cultivo.

---

## Inputs

| Input           | Tipo           | Origen                             |
| --------------- | -------------- | ---------------------------------- |
| `tabla_maestra` | `pd.DataFrame` | `data/feature_matrix.parquet` (M1) |

Columnas mínimas requeridas: `codigo_dane`, `cultivo`, `año`, `target_rendimiento`.

---

## Inputs esperados

| Columna              | Requerida | Regla                                                         |
| -------------------- | --------- | ------------------------------------------------------------- |
| `codigo_dane`        | Sí        | String de 5 dígitos, ya normalizado por M1                    |
| `cultivo`            | Sí        | `Café`, `Cacao` o `Maíz`                                      |
| `año`                | Sí        | Entero usado solo para orden temporal                         |
| `target_rendimiento` | Sí        | Float nullable; los nulos se excluyen del cálculo de etiqueta |

---

## Lógica / Proceso sugerido

1. Validar que `tabla_maestra` no esté vacía y contenga las columnas mínimas.
2. Verificar que no existan duplicados en la llave `codigo_dane + cultivo + año`; si aparecen, fallar porque la historia deja de ser ambigua.
3. Ordenar por `codigo_dane`, `cultivo` y `año`.
4. Para cada grupo `(codigo_dane, cultivo)`, calcular `promedio_historico` con `target_rendimiento` de filas anteriores al año actual.
5. Definir `umbral_riesgo = promedio_historico * 0.85`.
6. Asignar `target_riesgo = 1` si `target_rendimiento < umbral_riesgo`; de lo contrario `0`.
7. Si una fila no tiene historial suficiente para calcular `promedio_historico`, dejar `target_riesgo = NaN` y registrar warning para su exclusión posterior.

---

## Interfaz o contrato

La etapa debe devolver un DataFrame enriquecido con `target_riesgo` y, si se decide conservarlos para auditoría, con `promedio_historico` y `umbral_riesgo` como columnas auxiliares. Esas columnas auxiliares no deben consumirse como features en M3.2.

---

## Manejo de errores

| Situación                                     | Comportamiento                                              |
| --------------------------------------------- | ----------------------------------------------------------- |
| `tabla_maestra` vacía                         | `ValueError("[M3.1] tabla_maestra está vacía")`             |
| Faltan columnas clave                         | `ValueError` con lista explícita de columnas faltantes      |
| Llave duplicada `codigo_dane + cultivo + año` | `ValueError` con el grupo afectado                          |
| Ninguna fila tiene `target_riesgo` calculable | `ValueError` porque el conjunto no sirve para entrenar M3.2 |

---

## Salida esperada

DataFrame listo para M3.2 con una columna binaria `target_riesgo` y, opcionalmente, columnas auxiliares de auditoría.

---

## Caso feliz

- Dado un `tabla_maestra` válido
- Cuando se calcula la etiqueta con historial previo
- Entonces cada fila útil recibe `target_riesgo` sin usar información futura

---

## Validación

- Dado una fila con historial suficiente
- Cuando se evalúa la regla del 15%
- Entonces `target_riesgo` coincide con la comparación contra `promedio_historico * 0.85`
- Dado una fila sin historial suficiente
- Cuando se ejecuta la etapa
- Entonces la fila queda marcada con `NaN` y no rompe el proceso

---

## Criterios de aceptación

- [x] No se usa `año` futuro para calcular promedios históricos
- [x] La proporción de positivos se inspecciona y se deja trazada en logs o métricas internas
- [x] Las filas con `target_rendimiento` nulo no causan excepción y se excluyen del entrenamiento posterior

## Decisiones

- **Fecha:** 27 de mayo de 2026
- **Motivo:** Implementación del M3.1 para construcción del target de riesgo.
- **Decisiones:**
  - Se utilizó `transform` con `expanding(min_periods=1).mean().shift(1)` para evitar el uso de `inplace=True`, asegurando que no se toma información de años futuros y preservando vectorización por grupos (`codigo_dane`, `cultivo`).
  - Se asignan valores de forma predeterminada usando `np.where` manejando los valores `NaN` a los cálculos sin historial suficiente y se incluyeron logs para trazar la cantidad de nulos y la proporción de positivos según criterio de aceptación.

---

## Restricciones técnicas

- Usar transformaciones agrupadas en `Pandas` o `DuckDB`
- Mantener el cálculo reproducible por grupo y año
- Documentar la distribución de `target_riesgo` cuando se materialice el conjunto de entrenamiento
