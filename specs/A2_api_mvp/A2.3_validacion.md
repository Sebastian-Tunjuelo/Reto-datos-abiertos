## Agente

**Rol**: Eres un ingeniero de QA y MLOps encargado de certificar que la API A2 expone el contrato esperado y que su comportamiento observable coincide con la spec.

**Mentalidad**: Piensa como un auditor externo. No verificas cómo está implementada la lógica interna, sino que el contrato HTTP, los esquemas de respuesta y los errores observables sean correctos. Si un endpoint responde bien pero con una forma distinta a la acordada, la validación debe fallar.

**Restricciones**:

- No llamar funciones privadas de `modules/` para validar el contrato.
- No modificar archivos de datos ni modelos.
- Cada aserción debe tener un mensaje descriptivo con prefijo `❌`.
- Si una aserción falla, el script debe detenerse inmediatamente con `AssertionError`.
- El script debe poder ejecutarse de forma aislada.

---

## Objetivo

Verificar que los dos endpoints del bloque A2 funcionan como contrato público:

1. `GET /rendimiento/{municipio}/{cultivo}`
2. `GET /clima/{municipio}`

Además, comprobar que la normalización de entradas, el contrato de error y el orden temporal de las series se mantienen consistentes.

## Checklist de criterios de aceptación

### Disponibilidad básica

- [ ] `orchestrator.main.app` es importable
- [ ] `GET /openapi.json` responde y contiene los 2 endpoints de A2

### `GET /rendimiento/{municipio}/{cultivo}`

- [ ] Responde `200 OK` para una combinación válida
- [ ] Devuelve `codigo_dane`, `municipio`, `departamento`, `cultivo`, `n_anios`, `año_min`, `año_max` y `serie`
- [ ] `serie` es una lista no vacía
- [ ] Cada elemento de `serie` tiene exactamente `año`, `rendimiento`, `area_sembrada`, `rendimiento_prom3a` y `tendencia_3a`
- [ ] La serie está ordenada por `año` ascendente
- [ ] No hay años duplicados en la serie
- [ ] `rendimiento` y `area_sembrada` son numéricos o `null`
- [ ] `rendimiento_prom3a` y `tendencia_3a` son numéricos o `null`
- [ ] La consulta por nombre y por código DANE devuelve la misma entidad normalizada
- [ ] Un cultivo inválido responde `422`
- [ ] Un municipio inválido responde `404`
- [ ] Una combinación sin cobertura responde `404`

### `GET /clima/{municipio}`

- [ ] Responde `200 OK` para un municipio válido
- [ ] Devuelve `codigo_dane`, `municipio`, `departamento`, `n_anios`, `año_min`, `año_max` y `serie`
- [ ] `serie` es una lista no vacía
- [ ] Cada elemento de `serie` contiene las variables climáticas y anomalías acordadas en A2.2
- [ ] La serie está ordenada por `año` ascendente
- [ ] No hay años duplicados en la serie
- [ ] `prec_acum_mm` es numérico o `null`
- [ ] `hum_media_pct` está en el rango `[0, 100]` cuando no es `null`
- [ ] `anomalia_prec` y `anomalia_temp` están presentes en la respuesta
- [ ] La consulta por nombre y por código DANE devuelve la misma entidad normalizada
- [ ] Un municipio inválido responde `404`
- [ ] Una tabla climática ausente responde `503`

## Script de validación

Guardar como `specs/A2_api_mvp/validate_a2.py`.

El script debe:

- usar `fastapi.testclient.TestClient`
- importar `orchestrator.main.app`
- leer `data/eva_completa.parquet` y `data/clima_agregado.parquet` solo para seleccionar casos válidos o verificar disponibilidad
- validar OpenAPI, esquemas de respuesta, orden temporal y contrato de error
- evitar cualquier llamada a funciones privadas de `modules/`

---

## Caso feliz

- Dado un municipio válido del MVP y un cultivo válido.
- Cuando se invoca `GET /rendimiento/{municipio}/{cultivo}`.
- Entonces la respuesta devuelve una serie anual consistente, normalizada y ordenada.
- Dado un municipio válido del MVP.
- Cuando se invoca `GET /clima/{municipio}`.
- Entonces la respuesta devuelve el clima agregado anual consistente y normalizado.

---

## Validación negativa

- Dado un municipio fuera del MVP.
- Cuando se invoca cualquiera de los dos endpoints.
- Entonces la API responde con `404`.
- Dado un cultivo fuera del MVP.
- Cuando se invoca `GET /rendimiento/{municipio}/{cultivo}`.
- Entonces la API responde con `422`.
- Dado un artefacto de datos ausente.
- Cuando se ejecuta el script.
- Entonces el script falla con un mensaje claro y accionable.

---

## Criterios de aceptación

- Los dos endpoints están documentados en OpenAPI.
- El contrato de respuesta es estable y verificable.
- El orden temporal de las series es determinístico.
- El manejo de errores es consistente con la spec A2.
- La validación funciona de forma aislada.
