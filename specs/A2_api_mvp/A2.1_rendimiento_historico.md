# A2.1 - Serie histórica de rendimiento por municipio y cultivo

## Metadatos

| Campo         | Valor                                                                                                                                                           |
| ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ID            | A2.1                                                                                                                                                            |
| Módulo o área | `orchestrator/main.py` → `GET /rendimiento/{municipio}/{cultivo}`                                                                                               |
| Depende de    | `modules/agricultural/ingestion.py` (`load_eva_completa()`), `data/eva_completa.parquet`, `shared/config.py`, `shared/dane_codes.py`, `shared/normalization.py` |
| Produce para  | F2, F3                                                                                                                                                          |
| Estado        | ✅ Implementado                                                                                                                                                 |

## Agente

- role: Ingeniero backend / datos.
- goal: Convertir la historia EVA de un municipio y un cultivo en una serie anual estable, ordenada y reutilizable por el frontend.
- backstory: Esta capa alimenta la ficha municipal y el comparador. Debe ser determinística, no inventar años y mantener la misma regla de agregación que el feature engineering del proyecto.

## Objetivo

- Dado un municipio del MVP y un cultivo del MVP, devolver la serie anual de rendimiento y área sembrada con el mismo criterio de agregación usado por D5.2 cuando existan múltiples observaciones en un mismo año.

## Alcance

### Incluye

- `GET /rendimiento/{municipio}/{cultivo}` como endpoint read-only.
- Aceptar el municipio como nombre display o como código DANE.
- Normalizar cultivo a `Café`, `Cacao` o `Maíz`.
- Resolver el municipio a un código DANE canónico antes de consultar datos.
- Cargar `eva_completa.parquet` mediante `load_eva_completa()`.
- Filtrar por `codigo_dane` y `cultivo`.
- Agregar a nivel anual cuando haya más de una fila por `(codigo_dane, cultivo, año)`.
- Calcular `rendimiento_prom3a` y `tendencia_3a` usando solo observaciones previas.
- Devolver la serie ordenada por `año` ascendente.

### No incluye

- Predicción de futuro.
- SHAP.
- Escenarios contrafactuales.
- Chat o RAG.
- Escritura en base de datos.
- Interpolación, extrapolación o relleno artificial de años faltantes.

## Entradas

- `municipio` como nombre display o código DANE.
- `cultivo` como valor normalizable a `Café`, `Cacao` o `Maíz`.
- `data/eva_completa.parquet` como fuente histórica.

## Inputs esperados

| Campo       | Tipo  | Regla                                              |
| ----------- | ----- | -------------------------------------------------- |
| `municipio` | `str` | Nombre display del MVP o código DANE de 5 dígitos. |
| `cultivo`   | `str` | Debe resolverse a `Café`, `Cacao` o `Maíz`.        |

## Lógica / Proceso sugerido

1. Normalizar el municipio y el cultivo.
2. Resolver el municipio con las mismas reglas canónicas del proyecto.
3. Si el municipio no pertenece al MVP, devolver 404.
4. Si el cultivo no pertenece al MVP, devolver 422.
5. Cargar `eva_completa.parquet` y verificar que existan `codigo_dane`, `municipio`, `departamento`, `cultivo`, `año`, `rendimiento` y `area_sembrada`.
6. Filtrar la combinación `codigo_dane + cultivo` solicitada.
7. Si hay más de una observación para el mismo año, consolidarla con el mismo criterio usado por D5.2: rendimiento ponderado por `area_cosechada` cuando exista, o promedio simple en caso contrario; `area_sembrada` como suma.
8. Ordenar la serie por `año` ascendente.
9. Calcular `rendimiento_prom3a` con una ventana móvil sobre los años previos y `tendencia_3a` con la pendiente de los últimos tres años disponibles.
10. Si no quedan filas tras el filtro o la agregación, devolver 404 con un mensaje claro.
11. Construir una respuesta estable y repetible, sin usar datos futuros.

## Interfaz o contrato

### `GET /rendimiento/{municipio}/{cultivo}`

Respuesta `RendimientoHistoricoResponse`:

| Campo          | Tipo         | Nullable | Regla                                 |
| -------------- | ------------ | -------- | ------------------------------------- |
| `codigo_dane`  | `str`        | No       | 5 dígitos.                            |
| `municipio`    | `str`        | No       | Title Case.                           |
| `departamento` | `str`        | No       | Title Case.                           |
| `cultivo`      | `str`        | No       | `Café` \| `Cacao` \| `Maíz`.          |
| `n_anios`      | `int`        | No       | Número de años devueltos en la serie. |
| `año_min`      | `int`        | No       | Primer año disponible.                |
| `año_max`      | `int`        | No       | Último año disponible.                |
| `serie`        | `list[dict]` | No       | Serie anual ordenada ascendentemente. |

Cada elemento de `serie` debe contener:

| Campo                | Tipo    | Nullable | Regla                                      |
| -------------------- | ------- | -------- | ------------------------------------------ |
| `año`                | `int`   | No       | Año de la observación anual.               |
| `rendimiento`        | `float` | Sí       | t/ha estimadas o agregadas para ese año.   |
| `area_sembrada`      | `float` | Sí       | Ha totales del año.                        |
| `rendimiento_prom3a` | `float` | Sí       | Promedio móvil de los años previos.        |
| `tendencia_3a`       | `float` | Sí       | Pendiente de la ventana de 3 años previos. |

Ejemplo de respuesta:

```json
{
  "codigo_dane": "73168",
  "municipio": "Chaparral",
  "departamento": "Tolima",
  "cultivo": "Café",
  "n_anios": 18,
  "año_min": 2007,
  "año_max": 2024,
  "serie": [
    {
      "año": 2022,
      "rendimiento": 0.82,
      "area_sembrada": 1240.5,
      "rendimiento_prom3a": 0.79,
      "tendencia_3a": -0.03
    }
  ]
}
```

## Manejo de errores

| Situación                               | HTTP | Mensaje sugerido                                                     |
| --------------------------------------- | ---- | -------------------------------------------------------------------- |
| Municipio no reconocible                | 404  | `Municipio no encontrado en el MVP`                                  |
| Cultivo inválido                        | 422  | `Cultivo inválido. Valores permitidos: Café, Cacao, Maíz`            |
| No existe una fila histórica base       | 404  | `No hay datos históricos suficientes para la combinación solicitada` |
| La tabla EVA no está disponible         | 503  | `eva_completa.parquet no disponible`                                 |
| Faltan columnas mínimas en la tabla     | 503  | `eva_completa.parquet no disponible`                                 |
| Error inesperado al consolidar la serie | 500  | `Error interno al resolver la serie histórica de rendimiento`        |

## Salida esperada

- Una serie anual ordenada por `año`.
- Un único resultado por año.
- Valores agregados de forma determinística.
- `rendimiento_prom3a` y `tendencia_3a` calculados solo con observaciones previas.

## Caso feliz

- Dado `municipio = "Chaparral"` y `cultivo = "Café"`.
- Cuando el frontend invoca `GET /rendimiento/Chaparral/Café`.
- Entonces la API devuelve la serie anual histórica del municipio/cultivo con el nombre y el departamento normalizados.

## Validación

- Dado un municipio del MVP y un cultivo válido.
- Cuando se invoca el endpoint.
- Entonces la respuesta contiene la serie anual ordenada, sin años duplicados y con el esquema acordado.
- Dado un municipio válido consultado por nombre y por código DANE.
- Cuando se invoca el endpoint.
- Entonces ambas consultas devuelven la misma entidad normalizada.
- Dado un cultivo inválido.
- Cuando se invoca el endpoint.
- Entonces la API responde con 422.

## Criterios de aceptación

- [x] La serie se construye con datos EVA limpios, no con valores hardcodeados.
- [x] La agregación anual es consistente con D5.2 cuando un año tiene múltiples observaciones.
- [x] La respuesta no inventa años ni extrapola rendimiento.
- [x] El orden temporal es estable y reproducible.
- [x] El contrato es útil para frontend y auditable por QA.

## Restricciones técnicas

- No usar búsqueda fuzzy.
- No usar base de datos.
- No usar datos de otro cultivo para completar la serie.
- No exponer observaciones duplicadas por año.
- No recomputar la lógica de negocio de forma distinta a D5.2.
