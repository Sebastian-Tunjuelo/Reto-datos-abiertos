# A2.2 - Serie climática histórica por municipio

## Metadatos

| Campo         | Valor                                                                                                                                                          |
| ------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ID            | A2.2                                                                                                                                                           |
| Módulo o área | `orchestrator/main.py` → `GET /clima/{municipio}`                                                                                                              |
| Depende de    | `modules/climate/ingestion.py` (`load_clima_agregado()`), `data/clima_agregado.parquet`, `shared/config.py`, `shared/dane_codes.py`, `shared/normalization.py` |
| Produce para  | F2, F3                                                                                                                                                         |
| Estado        | ✅ Implementado                                                                                                                                                |

## Agente

- role: Ingeniero backend / datos.
- goal: Exponer la serie climática anual agregada de un municipio con un contrato estable para el frontend y la orquestación.
- backstory: Esta capa resume el clima ya procesado por D2.4. No debe volver a consultar IDEAM ni recalcular anomalías.

## Objetivo

- Dado un municipio del MVP, devolver la serie climática agregada anual con precipitación, temperatura, humedad, días secos, días lluviosos, conteos de estaciones y anomalías.

## Alcance

### Incluye

- `GET /clima/{municipio}` como endpoint read-only.
- Aceptar el municipio como nombre display o como código DANE.
- Resolver el municipio a un código DANE canónico antes de consultar datos.
- Cargar `clima_agregado.parquet` mediante `load_clima_agregado()`.
- Filtrar por `codigo_dane` y devolver únicamente la serie del municipio solicitado.
- Mantener las anomalías ya calculadas por D2.4.
- Ordenar la serie por `año` ascendente.

### No incluye

- Descarga de IDEAM crudo.
- Reagregación de observaciones meteorológicas.
- Recalcular anomalías.
- Predicción climática.
- SHAP, escenarios o chat.
- Escritura en base de datos.

## Entradas

- `municipio` como nombre display o código DANE.
- `data/clima_agregado.parquet` como fuente anual agregada.

## Inputs esperados

| Campo       | Tipo  | Regla                                              |
| ----------- | ----- | -------------------------------------------------- |
| `municipio` | `str` | Nombre display del MVP o código DANE de 5 dígitos. |

## Lógica / Proceso sugerido

1. Normalizar el municipio.
2. Resolver el municipio con las reglas canónicas del proyecto.
3. Si el municipio no pertenece al MVP, devolver 404.
4. Cargar `clima_agregado.parquet` y verificar que existan las columnas mínimas del contrato D2.
5. Filtrar por `codigo_dane`.
6. Ordenar la serie por `año` ascendente.
7. Si el municipio no tiene filas, devolver 404.
8. Devolver las variables ya agregadas, sin interpolar ni inventar años faltantes.
9. Mantener `anomalia_prec` y `anomalia_temp` tal como fueron calculadas por D2.4.

## Interfaz o contrato

### `GET /clima/{municipio}`

Respuesta `SerieClimaticaResponse`:

| Campo          | Tipo         | Nullable | Regla                                 |
| -------------- | ------------ | -------- | ------------------------------------- |
| `codigo_dane`  | `str`        | No       | 5 dígitos.                            |
| `municipio`    | `str`        | No       | Title Case.                           |
| `departamento` | `str`        | No       | Title Case.                           |
| `n_anios`      | `int`        | No       | Número de años devueltos en la serie. |
| `año_min`      | `int`        | No       | Primer año disponible.                |
| `año_max`      | `int`        | No       | Último año disponible.                |
| `serie`        | `list[dict]` | No       | Serie anual ordenada ascendentemente. |

Cada elemento de `serie` debe contener el esquema real de D2.4:

| Campo               | Tipo    | Nullable | Regla                                                |
| ------------------- | ------- | -------- | ---------------------------------------------------- |
| `año`               | `int`   | No       | Año de la observación.                               |
| `prec_acum_mm`      | `float` | Sí       | Precipitación acumulada anual.                       |
| `prec_dias_secos`   | `int`   | Sí       | Días con precipitación < 1 mm.                       |
| `prec_dias_lluvia`  | `int`   | Sí       | Días con precipitación >= 1 mm.                      |
| `temp_media_c`      | `float` | Sí       | Temperatura media anual.                             |
| `temp_max_media_c`  | `float` | Sí       | Proxy de temperatura máxima anual.                   |
| `hum_media_pct`     | `float` | Sí       | Humedad relativa media anual.                        |
| `n_estaciones_prec` | `int`   | No       | Estaciones usadas para precipitación.                |
| `n_estaciones_temp` | `int`   | No       | Estaciones usadas para temperatura.                  |
| `anomalia_prec`     | `float` | Sí       | Desviación fraccional respecto a la media histórica. |
| `anomalia_temp`     | `float` | Sí       | Desviación absoluta respecto a la media histórica.   |

Ejemplo de respuesta:

```json
{
  "codigo_dane": "73168",
  "municipio": "Chaparral",
  "departamento": "Tolima",
  "n_anios": 18,
  "año_min": 2007,
  "año_max": 2024,
  "serie": [
    {
      "año": 2022,
      "prec_acum_mm": 1890.4,
      "prec_dias_secos": 120,
      "prec_dias_lluvia": 245,
      "temp_media_c": 24.1,
      "temp_max_media_c": 31.8,
      "hum_media_pct": 78.6,
      "n_estaciones_prec": 6,
      "n_estaciones_temp": 5,
      "anomalia_prec": -0.08,
      "anomalia_temp": 0.4
    }
  ]
}
```

## Manejo de errores

| Situación                              | HTTP | Mensaje sugerido                                                   |
| -------------------------------------- | ---- | ------------------------------------------------------------------ |
| Municipio no reconocible               | 404  | `Municipio no encontrado en el MVP`                                |
| La tabla climática no está disponible  | 503  | `clima_agregado.parquet no disponible`                             |
| Faltan columnas mínimas en la tabla    | 503  | `clima_agregado.parquet no disponible`                             |
| Municipio válido sin filas en la tabla | 404  | `No hay datos climáticos disponibles para el municipio solicitado` |
| Error inesperado al resolver la serie  | 500  | `Error interno al resolver la serie climática`                     |

## Salida esperada

- Una serie anual ordenada por `año`.
- Un único resultado por año.
- Variables climáticas agregadas y anomalías preservadas tal como quedaron en D2.4.
- Sin interpolación, sin rellenar años faltantes y sin consultar IDEAM en tiempo de ejecución.

## Caso feliz

- Dado `municipio = "Chaparral"`.
- Cuando el frontend invoca `GET /clima/Chaparral`.
- Entonces la API devuelve la serie climática agregada anual del municipio con el nombre y el departamento normalizados.

## Validación

- Dado un municipio del MVP.
- Cuando se invoca el endpoint.
- Entonces la respuesta contiene la serie anual ordenada, sin años duplicados y con el esquema acordado.
- Dado un municipio consultado por nombre y por código DANE.
- Cuando se invoca el endpoint.
- Entonces ambas consultas devuelven la misma entidad normalizada.
- Dado un municipio inválido.
- Cuando se invoca el endpoint.
- Entonces la API responde con 404.

## Criterios de aceptación

- [x] La serie climática se sirve desde `clima_agregado.parquet`, no desde observaciones crudas.
- [x] El contrato preserva las anomalías y los conteos de estaciones del pipeline D2.
- [x] La respuesta es estable y reproducible para el mismo municipio.
- [x] No se inventan años, estaciones ni valores meteorológicos.
- [x] El contrato es útil para la ficha municipal y auditables por QA.

## Restricciones técnicas

- No usar búsqueda fuzzy.
- No usar base de datos.
- No volver a consultar IDEAM en runtime.
- No recomputar anomalías dentro del endpoint.
- No interpolar ni extrapolar series.
