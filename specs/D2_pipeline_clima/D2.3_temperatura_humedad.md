## Estrategia de filtrado — por municipio, no por codigoestacion

> ⚠️ **Cambio de arquitectura respecto al diseño original** — aplica a D2.3 igual que a D2.2

El servidor Socrata de `datos.gov.co` **no tiene índice sobre `codigoestacion`** en los datasets de observaciones (`sbwg-7ju4`, `uext-mhny`). Cualquier consulta con `WHERE codigoestacion IN (...)` provoca un full-scan y hace timeout, incluso con 1 sola estación y 1 año de datos.

**Solución adoptada**: filtrar por `municipio IN (...)` en MAYÚSCULAS, que sí está indexado. Las consultas se dividen en **ventanas de 6 meses** (`IDEAM_MONTH_CHUNK`, default 6 meses). Los resultados parciales se re-agregan localmente con pandas.

`df_estaciones` se acepta como parámetro por compatibilidad de interfaz pero no se usa para construir el filtro de observaciones.

En la corrida guardada ya se obtuvo temperatura y humedad reales para los municipios del MVP y ambos intermedios quedaron persistidos para depuración.

## Consultas SoQL

### Temperatura — media anual y máximo absoluto

```sql
-- Consulta principal (se repite por ventana mensual 2007-01 … 2024-12)
$select=municipio,
        date_trunc_y(fechaobservacion) AS año,
        avg(valorobservado) AS temp_media_c,
        max(valorobservado) AS temp_max_media_c,
        count(*) AS n_obs_temp
$where=municipio IN ('IBAGUE','CHAPARRAL','NEIVA','GARZON','PITALITO',
                     'SAN VICENTE DE CHUCURI','RIONEGRO','ANORI','AMALFI',
                     'PENSILVANIA','PALESTINA','VILLAVICENCIO',
                     'EL TAMBO','MIRANDA','VALLEDUPAR')
       AND valorobservado IS NOT NULL
       AND fechaobservacion >= '2024-01-01T00:00:00.000'
       AND fechaobservacion < '2024-02-01T00:00:00.000'
$group=municipio, date_trunc_y(fechaobservacion)
$order=municipio, date_trunc_y(fechaobservacion)
```

> `avg()` y `max()` son funciones de agregación válidas en SoQL. `temp_max_media_c` es el máximo absoluto del año — se usa como proxy de la media de máximas diarias, que requeriría una subconsulta no soportada en SoQL. Se documenta la limitación en el output.

> **Re-agregación local**: los resultados de múltiples ventanas mensuales para el mismo año se consolidan con `groupby(["municipio","año"]).agg({"temp_media_c": "mean", "temp_max_media_c": "max"})` en pandas.

### Humedad — media anual

```sql
$select=municipio,
        date_trunc_y(fechaobservacion) AS año,
        avg(valorobservado) AS hum_media_pct,
        count(*) AS n_obs_hum
$where=municipio IN (...) AND valorobservado IS NOT NULL
       AND fechaobservacion >= '...' AND fechaobservacion < '...'
$group=municipio, date_trunc_y(fechaobservacion)
$order=municipio, date_trunc_y(fechaobservacion)
```

## Lógica de transformación — `download_temperatura`

```
1. Recibir df_estaciones (output de D2.1) — validar que no esté vacío
   (no se usa para construir el filtro; se acepta por compatibilidad de interfaz)

2. Construir filtro SoQL con municipios en MAYÚSCULAS (no codigoestacion)
   Generar ventanas mensuales 2007-01 … 2024-12 (IDEAM_MONTH_CHUNK meses c/u)

3. Ejecutar consulta de temperatura para cada ventana mensual:
   fetch_all(dataset_id='sbwg-7ju4', select=..., where=..., group=..., order=...)
   Concatenar resultados → df_raw

4. Re-agregar df_raw por (municipio, año):
   temp_media_c → mean, temp_max_media_c → max, n_obs_temp → sum

5. Convertir columna 'año': pd.to_datetime(año_str).dt.year → int

6. Convertir columnas numéricas con pd.to_numeric(..., errors='coerce'):
   temp_media_c, temp_max_media_c, n_obs_temp

7. Filtrar valores físicamente imposibles:
   temp_media_c < 0 o > 45 → NaN; loggear municipio y año
   temp_max_media_c < 0 o > 50 → NaN; loggear

8. normalize_title_case(municipio)

9. Cruzar municipio → codigo_dane usando get_codigo()

10. Filtrar: conservar solo filas donde codigo_dane esté en codigos_objetivo

11. Filtrar años: conservar solo YEAR_START ≤ año ≤ YEAR_END

12. Ejecutar consulta n_estaciones_temp (misma estrategia de ventanas mensuales)
    Re-agregar por (municipio, año) con sum()
    Merge con df principal por (codigo_dane, municipio, año)

13. Loggear warning por cada municipio del MVP sin datos de temperatura
14. Retornar DataFrame con columnas:
    codigo_dane, municipio, año, temp_media_c, temp_max_media_c, n_estaciones_temp
```

## Lógica de transformación — `download_humedad`

```
1–11. Idéntico a download_temperatura, usando dataset 'uext-mhny'
      Re-agregación: hum_media_pct → mean

12. Filtrar valores físicamente imposibles:
    hum_media_pct < 0 o > 100 → NaN; loggear municipio y año

13. Filtrar años: conservar solo YEAR_START ≤ año ≤ YEAR_END

14. Loggear warning por cada municipio del MVP sin datos de humedad
15. Retornar DataFrame con columnas:
    codigo_dane, municipio, año, hum_media_pct
```

## Notas de implementación

### Función privada `_agregar_variable_climatica`

La lógica de los pasos 1–11 es idéntica entre `download_temperatura` y `download_humedad`. Se extrae a una función privada que delega a `_fetch_obs_por_municipio` (el helper central de ventanas mensuales):

```python
    dataset_id: str,
    df_estaciones: pd.DataFrame,
    select: str,
    codigos_objetivo: set[str],
    prefijo_log: str,
) -> pd.DataFrame:
    """Descarga y agrega una variable climática por municipio y año."""
```

Cada función pública construye su `select` específico y delega a `_agregar_variable_climatica`.

### `temp_max_media_c` — limitación documentada

SoQL no soporta subconsultas, por lo que no es posible calcular la media de máximas diarias directamente. Se usa `max(valorobservado)` como proxy (máximo absoluto del año). Esta limitación se documenta en el docstring de la función. Para el modelo predictivo, `temp_max_media_c` sigue siendo una señal útil de estrés térmico aunque no sea exactamente la media de máximas.

### Re-agregación local para temperatura

Como las ventanas mensuales devuelven `avg()` y `max()` parciales, la re-agregación correcta es:

- `temp_media_c`: promedio ponderado por `n_obs_temp` (o simplemente `mean` si los lotes son uniformes)
- `temp_max_media_c`: `max` de los máximos parciales

### Municipios con cobertura parcial

Igual que en D2.2: las filas faltantes quedan ausentes del DataFrame. D2.4 maneja los huecos.
